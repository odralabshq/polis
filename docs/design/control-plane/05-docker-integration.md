# Control Plane — Docker & Infrastructure Integration

## Dockerfile

Multi-stage build matching the existing toolbox pattern (`services/toolbox/Dockerfile`):

```dockerfile
# syntax=docker/dockerfile:1
FROM dhi.io/rust:1-dev@sha256:<pin> AS builder
WORKDIR /build
COPY services/control-plane/crates/ crates/
COPY services/control-plane/Cargo.toml services/control-plane/Cargo.lock* ./
COPY lib/crates/ ../../lib/crates/
RUN --mount=type=cache,target=/usr/local/cargo/registry,sharing=locked \
    --mount=type=cache,target=/usr/local/cargo/git,sharing=locked \
    --mount=type=cache,target=/build/target,sharing=locked \
    cargo build --release -p cp-server && \
    cp target/release/cp-server /cp-server

FROM dhi.io/debian-base:trixie@sha256:<pin>

COPY --from=builder /cp-server /usr/local/bin/cp-server

USER 65532

ENV RUST_LOG=info
ENV POLIS_CP_LISTEN_ADDR=0.0.0.0:9080
ENV POLIS_CP_VALKEY_URL=rediss://valkey:6379
EXPOSE 9080
CMD ["/usr/local/bin/cp-server"]
```

**Note:** Only `cp-server` is built into the Docker image. The TUI lives inside the `polis` CLI binary (see 03-tui.md).

## Docker Compose Service

Add to `docker-compose.yml`:

```yaml
  # =========================================================
  # CONTROL PLANE (Web UI + REST API)
  # =========================================================
  control-plane:
    build:
      context: .
      dockerfile: services/control-plane/Dockerfile
    image: ghcr.io/odralabshq/polis-control-plane-oss:${POLIS_CP_VERSION:-latest}
    container_name: polis-control-plane
    user: "65532:65532"
    read_only: true
    tmpfs:
      - /tmp:size=10M,mode=1777
    security_opt:
      - no-new-privileges:true
      - seccomp=./services/control-plane/config/seccomp.json
    cap_drop:
      - ALL
    deploy:
      resources:
        limits:
          memory: 256M
          cpus: "0.5"
    networks:
      gateway-bridge:
        ipv4_address: 10.30.1.30
    ports:
      - "127.0.0.1:9080:9080"
    environment:
      - RUST_LOG=info
      - POLIS_CP_LISTEN_ADDR=0.0.0.0:9080
      - POLIS_CP_VALKEY_URL=rediss://valkey:6379
      - POLIS_CP_VALKEY_USER=cp-server
      - POLIS_CP_VALKEY_PASS_FILE=/run/secrets/valkey_cp_server_password
    secrets:
      - valkey_cp_server_password
    volumes:
      - ./certs/valkey:/etc/valkey/tls:ro
    depends_on:
      state:
        condition: service_healthy
    healthcheck:
      test: ["CMD-SHELL", "timeout 2 sh -c 'exec 3<>/dev/tcp/localhost/9080' 2>/dev/null"]
      interval: 10s
      timeout: 5s
      retries: 3
      start_period: 15s
    restart: unless-stopped
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
        labels: "service"
    labels:
      service: "polis-control-plane"
```

### Key Decisions

- **Network:** `gateway-bridge` only — needs Valkey access, no workspace or internet access needed
- **IP:** `10.30.1.30` — unused in current topology (toolbox is .20, certgen is .7, sentinel is .5, gate is .6, resolver is .10)
- **Port:** `127.0.0.1:9080:9080` — bound to loopback only, preventing LAN exposure of the unauthenticated API
- **Secrets:** Dedicated `valkey_cp_server_password` — generated by `generate-secrets.sh` with a least-privilege ACL user
- **Volumes:** Same Valkey TLS certs as toolbox (`./certs/valkey:/etc/valkey/tls:ro`)
- **Security:** Matches toolbox hardening — non-root user 65532, read-only filesystem, all capabilities dropped, seccomp profile, memory limit

## Valkey ACL User

Add to `services/state/scripts/generate-secrets.sh`:

**Password generation:**
```bash
PASS_CP_SERVER="$(generate_password)"
HASH_CP_SERVER="$(hash_password "${PASS_CP_SERVER}")"
echo -n "$PASS_CP_SERVER" > "${OUTPUT_DIR}/valkey_cp_server_password.txt"
```

**ACL entry** (append to `valkey_users.acl`):
```
user cp-server on #${HASH_CP_SERVER} ~polis:blocked:* ~polis:approved:* ~polis:config:* ~polis:log:events -@all +GET +SET +SETEX +DEL +MGET +EXISTS +SCAN +ZADD +ZREVRANGE +ZCARD +ZREMRANGEBYRANK +PING
```

**Docker Compose secrets** (add to top-level `secrets:` block):
```yaml
  valkey_cp_server_password:
    file: ./secrets/valkey_cp_server_password.txt
```

**State service** (add to `state.secrets:` list):
```yaml
      - valkey_cp_server_password
```

## Seccomp Profile

Copy from `services/toolbox/config/seccomp.json` and place at `services/control-plane/config/seccomp.json`. The cp-server has the same syscall profile as the toolbox (network I/O, file reads, no privileged operations).

## Workspace Lints

`services/control-plane/Cargo.toml`:

```toml
[workspace]
members = [
    "crates/cp-server",
    "crates/cp-api-types",
]
resolver = "2"

[workspace.lints.rust]
unsafe_code = "deny"

[workspace.lints.clippy]
pedantic = { level = "warn", priority = -1 }
unwrap_used = "warn"
expect_used = "warn"
```

Matches the existing toolbox workspace lint configuration. The TUI code lives in the CLI crate, not in this workspace.

## Migration Path

The control plane coexists with the existing CLI security commands. The CLI continues to work via `docker exec polis-toolbox polis-approve`. In a future release, the CLI can be updated to call the control plane REST API instead, eliminating the `docker exec` pattern entirely:

```
# Before (current):
polis security pending → docker exec → polis-approve list-pending → Valkey

# After (future):
polis security pending → HTTP GET localhost:9080/api/v1/blocked → cp-server → Valkey
```

This migration is optional and can happen incrementally per command.

The new `polis dashboard` subcommand provides a live alternative to the one-shot `polis security` commands from day one.
